AgenticMemory 跨各种图形大小的核心操作的性能测量。所有基准测试都直接使用 Rust 引擎；对于 I/O 密集型操作，Python SDK 开销可以忽略不计，并且对于计算密集型操作，每个 FFI 调用会增加大约 5-15 微秒。

## 测试环境

|参数|价值|
|-----------|-------|
|硬件| Apple M4 Pro (ARM64)，64 GB 统一内存 |
|操作系统 | macOS（达尔文）|
|铁锈| 1.90.0（发布配置文件，`--release`）|
|基准框架| __AGENTRA_代码_0__ 0.5 |
|迭代|每次测量 100 次（最少），带有统计预热 |
|特征向量| 128 维，f32 |

所有基准测试均使用 `cargo bench` 使用发布模式编译和链接时优化来运行。结果代表预热后 100 次迭代的中值，置信区间为 95%。

## 按图形大小划分的详细结果

### 添加节点

是时候将单个认知事件（具有 200 个字符的内容和元数据的事实）插入到内存图中了。

|图形尺寸|中位数 |标准差 |笔记|
|------------|--------|---------|-------|
| 1K 节点 | 248纳秒| 12 纳秒 |缓存热，所有数据都适合 L1 |
| 10K 节点 | 276 纳秒 | 18 纳秒 |哈希图增长带来的边际增长 |
| 10万个节点 | 312 纳秒 | 25 纳秒 |偶尔调整哈希映射大小摊销 |

节点插入是 O(1) 摊销的。性能主要由节点 ID 的哈希映射插入和时间戳系统调用决定。

### 添加边缘

是时候插入一条边并更新源和目标邻接列表了。

|图形尺寸|中位数 |标准差 |笔记|
|------------|--------|---------|-------|
| 1K 个节点，5K 个边 | 0.9 毫秒 | 0.08 毫秒 |邻接表很小 |
| 10K 个节点，50K 个边 | 1.2 毫秒 | 0.11 毫秒 |每个节点平均 5 条边 |
| 100K 个节点，500K 个边 | 1.8 毫秒 | 0.15 毫秒 |邻接表增长 |

边插入对于边本身来说是 O(1) 分摊的，对于邻接表管理来说是 O( Degree ) 。与节点插入相比，更高的绝对时间来自于更新两个邻接列表并验证两个端点。

### 图遍历（BFS）

从单个起始节点开始进行广度优先遍历的时间。图的平均度为 10（每个节点 5 个传出边、5 个传入边）。

|图形尺寸|深度 3 |深度 5 |深度 7 |
|------------|---------|---------|---------|
| 1K 节点 | 0.4 毫秒 | 0.8 毫秒 | 1.1 毫秒 |
| 10K 节点 | 1.2 毫秒 | 3.4 毫秒 | 8.7 毫秒 |
| 10万个节点 | 3.1 毫秒 | 12.4 毫秒 | 45.2 毫秒 |

遍历时间取决于访问的节点数量，该数量随着深度（受图大小和度数限制）呈指数增长。访问集检查（哈希集查找）是每个节点的主要成本。

### 相似性搜索

对所有 128 维特征向量进行强力余弦相似度搜索。返回前 k 个结果。

|图形尺寸|前 5 名 |前 10 名 |前 50 名 |
|------------|-------|--------|--------|
| 1K 节点 | 0.9 毫秒 | 1.0 毫秒 | 1.1 毫秒 |
| 10K 节点 | 8.2 毫秒 | 9.0 毫秒 | 9.4 毫秒 |
| 10万个节点 | 82 毫秒 | 84 毫秒 | 87 毫秒 |

相似性搜索的时间复杂度为 O(N * D)，其中 N 是节点数，D 是向量维度。连续向量布局支持 SIMD 自动向量化——编译器在 ARM 上生成 NEON 指令，每个周期处理 4 个浮点数。 top-k 选择增加了最小的开销（二叉堆，O(N log k)）。

在 100K 节点上，集群映射索引（启用后）通过仅扫描相关集群，将搜索时间减少到大约 15-20 毫秒。

### 文件写入

是时候将完整的图序列化为 `.amem` 文件了，包括 LZ4 内容压缩和向量块构建。

|图形尺寸|中位数 |文件大小 |吞吐量|
|------------|--------|-----------|------------|
| 1K 个节点，5K 个边 | 4.1 毫秒 | 0.7 MB | 170MB/秒|
| 10K 个节点，50K 个边 | 32.6 毫秒 | 7.1 MB | 218 MB/秒 |
| 100K 个节点，500K 个边 | 310 毫秒 | 71 MB | 229 MB/秒 |

写入时间主要由内容块的 LZ4 压缩和顺序写入决定。该格式的顺序布局（写入期间无随机查找）可最大限度地提高吞吐量。

### 文件读取

是时候将 `.amem` 文件反序列化到内存图中，包括 LZ4 解压缩。

|图形尺寸|中位数 |文件大小 |吞吐量|
|------------|--------|-----------|------------|
| 1K 节点 | 0.5 毫秒 | 0.7 MB | 1.4 GB/秒 |
| 10K 节点 | 3.7 毫秒 | 7.1 MB | 1.9 GB/秒 |
| 10万个节点 | 34 毫秒 | 71 MB | 2.1 GB/秒 |

读取性能受益于 LZ4 的快速解压缩（>3 GB/s）和顺序文件布局。吞吐量随着大小的增加而增加，因为固定开销（标头解析、分配）被摊销。

### 内存映射读取 (MmapReader)

是时候打开文件并通过内存映射 I/O 访问单个随机节点了。

|图形尺寸|打开|随机节点访问 |笔记|
|------------|------|--------------------|-------|
| 1K 节点 | 0.1 毫秒 | 0.3 我们 |页面缓存中的整个文件 |
| 10K 节点 | 0.1 毫秒 | 0.4 我们 |页面缓存中的整个文件 |
| 10万个节点 | 0.2 毫秒 | 0.5 我们 |可能会触发页面错误 |

内存映射访问避免预先读取整个文件。节点访问是初始页面错误后的直接指针取消引用。这对于从大大脑中读取一小部分节点的应用程序来说是理想的选择。

## 比较上下文

这些基准测试旨在了解 AgenticMemory 的性能概况。与其他系统的直接比较需要仔细的方法，因为它们解决不同的问题。

**与矢量数据库的主要架构差异：**
- AgenticMemory 是一个嵌入式库，而不是客户端-服务器系统。没有网络往返。
- 纯矢量数据库中不存在图结构（边、遍历）。
- 二进制文件格式针对单写入器工作负载进行了优化，而不是针对并发多写入器访问。

**与图数据库（Neo4j 等）的主要架构差异：**
- AgenticMemory 是一个文件，而不是服务器。没有查询语言解析，没有事务管理。
- 固定大小的节点记录和连续布局比指针密集型图形表示更适合缓存。
- 代价是灵活性较低：没有临时查询、没有架构迁移、没有 ACID 事务。

## 重现基准测试

### 先决条件

```bash
# Rust toolchain
rustup update stable

# Clone and build
git clone https://github.com/anthropic/agentic-memory.git
cd agentic-memory
```

### 运行全部 基准测试

```bash
cargo bench
```

结果通过 HTML 报告写入 `target/criterion/`，包括统计分析、吞吐量图表和回归检测。

### 运行特定基准测试

```bash
# Only node operations
cargo bench -- add_node

# Only I/O benchmarks
cargo bench -- file_write
cargo bench -- file_read

# Only search benchmarks
cargo bench -- similarity_search
```

### 生成报告

```bash
cargo bench -- --save-baseline my_hardware
```

`target/criterion/report/index.html` 处的 HTML 报告包括：
- 每个基准的分布图
- 平均值、中位数和标准差
- 吞吐量计算
- 与之前运行相比的变化检测

### 自定义图形大小

基准测试接受环境变量来控制图形大小：

```bash
BENCH_NODES=50000 BENCH_EDGES=250000 cargo bench
```

### 分析

如需详细分析，请使用 `cargo-flamegraph`：

```bash
cargo install flamegraph
cargo flamegraph --bench core_benchmarks -- --bench
```

这会生成一个 SVG 火焰图，显示基准测试执行期间花费的时间。

### 性能等级

查询在 100K 个节点上分为三层：

- **Interactive (\<100 ms):** BM25, hybrid, PageRank, degree, BFS, Dijkstra, belief revision, drift -- suitable for per-query use during conversations
- **周期性（1-60 秒）：** 介数中心性、整合——每个会话或按计划运行一次
- **Offline (>60 s):** Gap detection, analogical reasoning -- designed for batch analysis of large graphs; both complete in \<3s at 10K nodes

### BM25 指数加速

|图形尺寸|快速路径（术语索引）|慢速路径（全面扫描）|加速|
|------------|----------------------|-----------------------|---------|
| 10K 节点 | 186 我们 | 8.59 毫秒 | 46x |
| 10万个节点 | 1.58 毫秒 | 122 毫秒 | 77x |

倒排索引加速随着图的大小而增长，因为快速路径成本取决于发布列表大小（n 中的次线性），而慢速路径始终为 O(n)。
