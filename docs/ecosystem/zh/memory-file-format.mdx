本文档描述了 `.amem` 文件的二进制布局——AgenticMemory 大脑数据的磁盘格式。该格式专为快速随机访问、内存映射 I/O 和紧凑存储而设计。

## 设计目标

- **零拷贝读取：** 节点和边缘记录是固定大小的，无需反序列化即可直接访问内存。
- **内存映射友好：** 布局针对 `mmap()` 访问进行对齐。操作系统处理分页，因此大于可用 RAM 的大脑文件可以有效地工作。
- **紧凑：** 内容经过 LZ4 压缩。特征向量存储为连续的浮点数组。没有 JSON 开销，每条记录没有重复的字段名称。
- **原子写入：** 最后写入更新标头。写入过程中的崩溃会使之前的有效状态保持不变。
- **向前兼容：** 版本字段和保留的标头字节允许格式演变，而不会破坏现有的阅读器。

## 总览

`.amem` 文件由六个连续的部分组成：

```
+------------------+
|   Magic + Header |  64 bytes
+------------------+
|   Node Records   |  node_count * 64 bytes
+------------------+
|   Edge Records   |  edge_count * 13 bytes
+------------------+
|   Content Block  |  variable (LZ4 compressed)
+------------------+
|  Feature Vectors |  node_count * dimension * 4 bytes
+------------------+
|     Indexes      |  variable
+------------------+
```

## 第 1 部分：标头（64 字节）

文件头占据文件的前 64 个字节。

|偏移|尺寸|类型 |领域 |描述 |
|--------|------|------|-------|-------------|
| 0 | 4 | `[u8; 4]` | `magic` |魔术字节：`0x41 0x4D 0x45 0x4D`（ASCII“AMEM”）。 |
| 4 | 2 | `u16` | `version` |格式版本。当前：`1`。 |
| 6 | 2 | `u16` | `flags` |位域。位 0：有向量。位 1：有索引。位 2：内容已压缩。 |
| 8 | 4 | `u32` | `node_count` |节点记录总数。 |
| 12 | 12 4 | `u32` | `edge_count` |边缘记录总数。 |
| 16 | 16 2 | `u16` | `dimension` |特征向量维数。默认值：128。
| 18 | 18 2 | `u16` | `session_count` |不同会话的数量。 |
| 20 | 8 | `u64` | `content_offset` |到内容块开头的字节偏移量。 |
| 28 | 28 8 | `u64` | `content_length` |内容块的长度（以字节为单位）（压缩后）。 |
| 36 | 36 8 | `u64` | `vector_offset` |特征向量块开头的字节偏移量。 |
| 44 | 44 8 | `u64` | `index_offset` |到索引块开头的字节偏移量。 |
| 52 | 52 4 | `u32` | `content_uncompressed` |内容块的未压缩大小。 |
| 56 | 56 8 | `[u8; 8]` | `reserved` |保留以供将来使用。必须为零。 |

**验证规则：**
- `magic` 必须是 `0x414D454D`。
- 对于此规范，`version` 必须是 `<= 1`。
- `node_count` 和 `edge_count` 必须与文件大小一致。
- `dimension` 必须是正整数（通常为 128）。

## 第 2 部分：节点记录

紧接着标题。每个节点记录都是固定的64字节。

|偏移|尺寸|类型 |领域 |描述 |
|--------|------|------|-------|-------------|
| 0 | 1 | `u8` | `event_type` |事件类型枚举：0=事实、1=决策、2=推理、3=纠正、4=技能、5=情节。 |
| 1 | 3 | `[u8; 3]` | `padding` |对齐填充。必须为零。 |
| 4 | 4 | `u32` | `session` |会话 ID。 |
| 8 | 4 | `f32` | `confidence` |置信度分数（IEEE 754 单精度）。 |
| 12 | 12 8 | `i64` | `timestamp` |以秒为单位的 Unix 时间戳 (UTC)。 |
| 20 | 8 | `u64` | `content_offset` |解压缩内容块内此节点内容开始的偏移量。 |
| 28 | 28 4 | `u32` | `content_length` |该节点内容的长度（以字节为单位）（解压缩）。 |
| 32 | 32 8 | `u64` | `vector_offset` |矢量块内的偏移。如果不存在向量，则设置为 `u64::MAX`。 |
| 40| 8 | `u64` | `metadata_offset` | JSON 编码元数据的内容块内的偏移量。 `u64::MAX` 如果没有元数据。 |
| 48 | 48 4 | `u32` | `metadata_length` |元数据的长度（以字节为单位）。如果没有元数据，则为 0。 |
| 52 | 52 12 | 12 `[u8; 12]` | `reserved` |预订的。必须为零。 |

**总计：** 每个节点 64 字节。

**笔记：**
- 节点 ID 是隐式的 - 节点 N 位于距文件开头的偏移量 `64 + (N * 64)` 处。
- `content_offset` 和 `metadata_offset` 是 **解压缩** 内容块的偏移量，而不是原始文件的偏移量。
- `vector_offset` 是向量块部分的字节偏移量。

## 第 3 部分：边缘记录

紧接着节点记录。每个边缘记录都是固定的13字节。

|偏移|尺寸|类型 |领域 |描述 |
|--------|------|------|-------|-------------|
| 0 | 4 | `u32` | `source` |源节点ID。 |
| 4 | 4 | `u32` | `target` |目标节点ID。 |
| 8 | 1 | `u8` | `edge_type` |边类型枚举：0=CausedBy、1=Supports、2=Contradicts、3=Supersedes、4=RelatedTo、5=PartOf、6=TemporalNext。 |
| 9 | 4 | `f32` | `weight` |边缘权重（IEEE 754 单精度）。 |

**总计：** 每条边 13 个字节。

**笔记：**
- 边按源节点 ID 排序，以实现高效的邻接查找。
- `source` 和 `target` 都必须是有效的节点 ID（小于 `node_count`）。

## 第 4 部分：内容块

包含所有节点内容和元数据字符串的单个 LZ4 压缩块，端到端连接。

**结构（解压后）：**

```
[node_0_content][node_1_content]...[node_N_content][node_0_metadata][node_1_metadata]...
```

所有内容均采用 UTF-8 编码。元数据是 JSON 编码的 UTF-8（具有字符串键和字符串值的平面对象）。

**压缩：**
- 算法：LZ4（帧格式）。
- 标头字段 `content_length` 存储压缩大小。
- 标头字段 `content_uncompressed` 存储解压后的大小。
- 如果未设置 `flags` 位 2，则内容块将以未压缩的方式存储（对于非常小的大脑，其中压缩开销不值得）。

**LZ4 的基本原理：** LZ4 解压缩以内存带宽速度（通常为 3--5 GB/s）运行，与 I/O 相比，它实际上是免费的。自然语言文本的压缩比通常为 2--3 倍。

## 第 5 节：特征向量

IEEE 754 单精度浮点值的连续块。每个节点的向量是 `dimension` 浮点数（默认值：128 个浮点数 = 每个向量 512 字节）。

**布局：**

```
[node_0_vector: f32 * dimension][node_1_vector: f32 * dimension]...[node_N_vector]
```

**笔记：**
- 向量按节点 ID 顺序存储。
- 如果节点没有向量（向量偏移量为 `u64::MAX`），则相应的槽包含全零。
- 连续布局对于 SIMD 加速的相似性搜索至关重要 - CPU 可以扫描向量而无需指针追踪。
- 总大小：`node_count * dimension * 4` 字节。

## 第 6 节：索引

索引部分包含用于加速查询的辅助数据结构。仅当设置了 `flags` 位 1 时，它才存在。

### 类型索引（位图）

将事件类型映射到节点 ID 的位图索引。每个事件类型都有一个由 `node_count` 位组成的位集，其中如果节点 N 具有该事件类型，则设置位 N。

|偏移|尺寸|类型 |描述 |
|--------|------|------|-------------|
| 0 | 4 | `u32` | `index_type`：0x01（类型索引）。 |
| 4 | 4 | `u32` | `num_types`：事件类型数量 (6)。 |
| 8 |变化 | `[u8; ceil(node_count / 8)] * num_types` |打包位集，每种事件类型一个。 |

### 会话索引

将会话 ID 映射到其组成节点 ID 范围。

|偏移|尺寸|类型 |描述 |
|--------|------|------|-------------|
| 0 | 4 | `u32` | `index_type`：0x02（会话索引）。 |
| 4 | 4 | `u32` | `num_sessions`：会话数。 |
| 8 |变化 | `[(session_id: u32, start_node: u32, end_node: u32)] * num_sessions` |会话到节点范围的映射。 |

### 时间索引

用于高效时间范围查询的（时间戳、node_id）对的排序数组。

|偏移|尺寸|类型 |描述 |
|--------|------|------|-------------|
| 0 | 4 | `u32` | `index_type`：0x03（时间索引）。 |
| 4 | 4 | `u32` | `num_entries`：条目数。 |
| 8 |变化 | `[(timestamp: i64, node_id: u32)] * num_entries` |按时间戳升序排序。 |

### 聚类图

用于近似最近邻搜索的预先计算的簇分配。节点根据其特征向量分组为簇。

|偏移|尺寸|类型 |描述 |
|--------|------|------|-------------|
| 0 | 4 | `u32` | `index_type`：0x04（集群映射）。 |
| 4 | 4 | `u32` | `num_clusters`：簇数。 |
| 8 | 4 | `u32` | `dimension`：向量维度。 |
| 12 | 12变化 | `[f32; dimension] * num_clusters` |聚类质心向量。 |
|变化 |变化 | `[(cluster_id: u32, node_id: u32)] * node_count` |节点到集群的分配，按 cluster_id 排序。 |

## 版本兼容性

### 版本 1（当前）

本文档中描述的初始版本格式。

**读者必须：**
- 拒绝 `magic != 0x414D454D` 的文件。
- 拒绝 `version > 1` 的文件。
- 优雅地处理向量的缺失（标志位 0 未设置）。
- 通过回退到线性扫描来处理索引的缺失（标志位 1 未设置）。

### 前向兼容性

未来的版本可能：
- 添加新的事件类型（`event_type` 字段中的值 >= 6）。老读者应该将未知类型视为不透明。
- 添加新的边缘类型（`edge_type` 字段中的值 >= 7）。老读者应该将未知类型视为不透明。
- 添加新的索引类型。老读者应该跳过未知的索引类型。
- 增加标题大小。老读者应该使用 `content_offset` 来查找内容块而不是硬编码偏移量。
- 将保留的标头字节用于新字段。

### 字节顺序

所有多字节整数和浮点数都以 **little-endian** 字节顺序存储。

## 尺寸估计

对于具有 N 个节点、M 个边和 128 维向量的大脑：

|组件|尺寸|
|-----------|------|
|标题 | 64 字节 |
|节点记录| N * 64 字节 |
|边缘记录| M * 13 字节 |
|内容块 | ~40% of raw content size (LZ4) |
|特征向量| N * 512 字节 |
|索引 | ~N * 20 字节（大约）|

**示例：** 具有 100,000 个节点、500,000 个边、每个节点平均内容为 200 字节的大脑：

|组件|尺寸|
|-----------|------|
|标题 | 64 乙 |
|节点记录| 6.1 MB |
|边缘记录| 6.2 MB |
|内容块 | ~8 MB（20 MB 原始数据，~2.5 倍压缩）|
|特征向量| 48.8 MB |
|索引 | 〜1.9 MB |
| **总计** | **~71 MB** |
