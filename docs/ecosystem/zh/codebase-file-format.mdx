`.acb` 二进制格式将完整的代码概念图存储在单个文件中。本文档描述了磁盘布局、部分结构和设计原理。

## 设计目标

1. **O(1) 随机访问。** 通过 ID 查找任何代码单元，无需扫描文件。
2. **紧凑。** 用于可变长度字符串的 LZ4 压缩。单位和边的固定大小记录。
3. **内存可映射。** 该格式支持 `mmap()` 对单元和边缘表进行零拷贝访问。
4. **向前兼容。** 新字段附加到标头。老读者会跳过未知的部分。

## 文件布局

```
Offset 0x00   ┌─────────────────────────────┐
              │         Header (128 B)       │
              ├─────────────────────────────┤
              │     Unit Table (96N bytes)    │  N = unit_count
              ├─────────────────────────────┤
              │     Edge Table (40M bytes)    │  M = edge_count
              ├─────────────────────────────┤
              │  String Pool (LZ4 compressed) │  Variable size
              ├─────────────────────────────┤
              │  Feature Vectors (f32 array)  │  N * dim * 4 bytes
              └─────────────────────────────┘
```

| 0x04 | 0x04 4 | `u32` |版本 |格式版本（当前为 1） |
| 0x08 | 0x08 8 | `u64` |单位计数 |代码单元数量|
| 0x10 | 0x10 8 | `u64` |边数 |边数 |
| 0x18 | 0x18 8 | `u64` |字符串池偏移量字符串池部分的字节偏移量 |
| 0x20 | 0x20 8 | `u64` |字符串池大小 |字符串池的压缩大小|
| 0x28 | 0x28 8 | `u64` |特征偏移 |特征向量部分的字节偏移量|

## 边缘表

从单位表之后开始。每个边缘记录为 40 字节。

|偏移|尺寸|类型 |领域 |描述 |
|:---|:---|:---|:---|:---|
| 0x00 | 0x00 8 | `u64` |源 ID |源单位 ID |
| 0x08 | 0x08 8 | `u64` |目标 ID |目标单位 ID |
| 0x10 | 0x10 1 | `u8` |边缘类型 | EdgeType 枚举判别式 |
| 0x11 | 0x11 7 | `[u8; 7]` |填充|对齐填充|
| 0x18 | 0x18 8 | `f64` |重量 |边权重 (0.0 - 1.0) |
| 0x20 | 0x20 8 | `[u8; 8]` |保留 |保留 |

总计：每条边 40 字节。

## 字符串池

字符串池包含所有可变长度文本：单元名称、限定名称、文件路径、签名和文档摘要。存储为单个连续缓冲区，LZ4 压缩。

读取时，整个池被解压缩到内存中。单元记录中的字符串引用使用（偏移量、长度）对进入此解压缩的缓冲区。

＃＃＃ 压缩

LZ4 块压缩用于字符串池。源代码元数据的典型压缩比：

- 英文标识符：~2.5 倍压缩
- 具有公共前缀的文件路径：~3-4x 压缩
- 文档文本：~2-3 倍压缩

LZ4 解压缩在现代硬件上的运行速度为 3-5 GB/s，使得解压缩成本可以忽略不计。

## 特征向量

特征向量存储为 `f32` 值的平面数组，每个单元一个向量。单元 N 的向量从偏移量 `feature_offset + N * dimension * 4` 开始。

默认维度为 64，可在编译时配置。由于 f32 值压缩效果不佳，因此向量未压缩。

## 版本控制

标头中的 `version` 字段启用前向兼容性：

- **版本 1**（当前）：本文档中描述的基本格式。
- 未来版本将通过在现有部分之后附加新部分并使用保留的标头字段来保持向后兼容性。

读者应该检查版本字段并拒绝版本不受支持的文件，而不是尝试解析未知格式。

## 校验和

当前格式不包括校验和。可以使用外部工具（例如 `blake3sum`）验证文件完整性。未来版本中可能会使用保留的标头空间添加校验和字段。
